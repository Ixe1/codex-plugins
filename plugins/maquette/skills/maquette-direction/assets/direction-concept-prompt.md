Create a greenfield website direction concept.

Goal:
- Explore a concrete page direction before the brand kit is locked.
- Show enough real website structure to judge hierarchy, tone, density, component needs, and asset needs.
- Treat the result as a directional seed, not a final page implementation contract.

Show:
- header and primary navigation direction
- hero hierarchy and primary CTA treatment
- one or two representative content sections
- at least one reusable component family, such as feature cards, pricing cards, testimonials, stats, forms, product cards, data previews, or editorial modules
- footer or terminal-region direction when relevant
- visual cues for responsive behavior, especially navigation and stacking, when the concept includes primary navigation
- likely raster asset needs, such as hero imagery, product imagery, lifestyle/story imagery, app/device imagery, background textures, or generated decorative rasters

Creative constraints:
- no logo generation, no wordmarks, no brand marks, no mascots, no badges, no seals, no app icons, no emblems, no monograms, and no trademark-like elements
- do not turn the brand or product name into a logo-like masthead or mark
- use neutral temporary product text if needed
- prioritize a coherent visual direction over exhaustive page detail
- make the hierarchy, section density, visual tone, and component families easy to inspect at normal preview size
- avoid clutter, tiny labels, and ambiguous decorative-only layouts

Handoff requirements:
- make durable brand candidates visible: palette direction, type personality, spacing rhythm, radius style, surface treatment, shadow/elevation language, and state/focus personality
- make page-local flourishes identifiable so they can stay local rather than becoming global tokens
- make accessibility risks visible where possible, especially contrast, small text, tap targets, and complex imagery behind text
- make asset and implementation feasibility clear enough for a later direction inventory

Do not create final production page code from this image. The selected concept will be normalized into a brand board, design-system tokens, an approved page concept, section-level fidelity contracts, and then a final page implementation. Component contracts are optional backfill unless the user asks for a reusable design system or complex reusable interactions.
